Practical Reference

Accessibility Guide

A practical WCAG 2.1 AA guide for Storyline 360 development — built from real project experience, not the spec document. Seven sections plus an interactive pre-publish checklist.

01

Focus Order — The First Thing to Get Right

Storyline assigns tab order based on the sequence objects were created on the slide — not their visual position. Draw a button first, then a heading, then another button, and keyboard users tab through them in creation order, not reading order. This is the single most common accessibility failure in Storyline courses, and the easiest to prevent if addressed at the template stage.

The Requirement WCAG 2.4.3 — Focus Order
Tab order must follow a logical, predictable sequence — typically top-left to bottom-right, matching the visual reading order of the slide.
Set explicitly, every slide: Home → Tab Order in Storyline. Don't rely on the default creation-order sequence.
Decorative elements excluded: Background shapes, dividers, and purely visual elements should not appear in the tab sequence at all.
Why This Has to Happen First Template-Level Fix
Tab order is the accessibility requirement most expensive to retrofit. Fixing it on a finished 40-slide course means opening every slide individually. Fixing it at the master slide / template stage — before content exists — costs almost nothing. This is why focus order is Section 01: every other accessibility decision in this guide is easier when the foundation is right first.
✓ Correct Approach
  • Set tab order explicitly before adding content to a new template
  • Order follows visual layout: title → body → primary action → secondary action
  • Re-check tab order after moving or resizing any object
  • Exclude decorative shapes and dividers from the sequence entirely
✗ Common Failure
  • Relying on Storyline's default creation-order tab sequence
  • Tab order jumps: button → title → second button → image
  • Forgetting to re-verify after a late-stage layout change
  • Decorative elements stealing a tab stop from real content
02

Colour Contrast — The Numbers That Matter

WCAG 2.1 Level AA requires a contrast ratio of at least 4.5:1 for normal text and 3:1 for large text (18pt or 14pt bold). Most Storyline templates fail this standard. "Dark text on light background" is not enough — you need to measure the actual ratio.

The Minimum Requirements WCAG 1.4.3 — Level AA
Normal text (under 18pt / under 14pt bold): Minimum 4.5:1 contrast ratio
Large text (18pt+ / 14pt+ bold): Minimum 3:1 contrast ratio
UI components and icons: Minimum 3:1 against adjacent colours
Decorative text in images: No requirement (but avoid putting critical info in images)
Disabled controls: No requirement (but make it visually clear they're disabled)
Tools for Checking Contrast Free Tools
WebAIM Contrast Checker — webaim.org/resources/contrastchecker — paste hex codes, get instant ratio
Colour Contrast Analyser — TPGi app — pipette any colour from your screen
Chrome DevTools — hover over any element in inspector, contrast ratio appears in tooltip
axe DevTools — browser extension that audits an entire page automatically
✓ Colour Pairs That Pass
  • #E4E8F1 on #080C18 — 13.2:1 (AAA)
  • #F0B429 on #080C18 — 7.2:1 (AAA)
  • #2FCCA1 on #080C18 — 6.1:1 (AA)
  • #000000 on #FFFFFF — 21:1 (Maximum)
  • #FFFFFF on #1A6CF0 — 4.7:1 (AA)
✗ Colour Pairs That Fail
  • #777777 on #AAAAAA — 1.8:1 (Fail)
  • #4B4B4B on #282828 — 1.7:1 (Fail)
  • Grey text on white without measuring
  • Light blue on white (common Storyline default)
  • Yellow on white — almost always fails
03

Alt Text Strategy — Not Just "Add Alt Text"

Alt text is not just about adding a description to every image. The strategy depends on whether the image is informational, decorative, complex, or functional. Wrong alt text is sometimes worse than missing alt text — it creates confusion and noise for screen reader users.

The Four Types of Images WCAG 1.1.1 — Level A
1. Informational: Conveys information necessary to understand the content → Must have descriptive alt text that conveys the same information
2. Decorative: Only for visual effect, adds no information → Set alt="" (empty) to be skipped by screen readers. In Storyline: uncheck "Object is visible to accessibility tools"
3. Functional: A button or link with an image → Alt text should describe the function, not the image ("Submit form", not "Arrow pointing right")
4. Complex: Charts, graphs, diagrams → Short alt text plus a long description — or equivalent text on the slide itself
✓ Good Alt Text Examples
  • Chart: "Bar chart showing 40% increase in errors after training gap"
  • Workflow: "5-step compliance process: Identify → Assess → Report → Document → Review"
  • Person: "Doctor reviewing patient chart in hospital setting"
  • Button with icon: "Submit quiz answers"
  • Decorative swirl: alt="" (empty — correctly excluded)
✗ Bad Alt Text Examples
  • "image" or "graphic" — meaningless to screen reader users
  • "photo1.jpg" — file name as alt text
  • Repeating the adjacent caption word-for-word
  • Adding alt text to purely decorative images (creates noise)
  • "Click here button" for a button — redundant word "button"
Setting Alt Text in Storyline Step by Step
For informational images: Right-click → Accessibility → type in "Alt Text" field
For decorative images: Right-click → Accessibility → uncheck "Object is visible to accessibility tools"
For complex charts: Add alt text AND provide equivalent text on the slide or in an expandable section
Audit tool: Publish to HTML5 → open in browser → run axe DevTools extension → it will flag every missing alt text
04

Keyboard Navigation — Every Interaction

All interactive elements in a Storyline course must be operable by keyboard alone — Tab to focus, Enter or Space to activate, Escape to close. Drag-and-drop, click-and-reveal, and custom interactions must have keyboard equivalents. This is the hardest accessibility requirement to retrofit — it must be designed in from the start.

Required Keyboard Behaviours WCAG 2.1.1 — Level A
Tab: Move focus to next focusable element
Shift + Tab: Move focus to previous focusable element
Enter / Space: Activate focused button, link, or control
Escape: Close modal, popup, or overlay layer
Arrow keys: Navigate within a component (radio buttons, sliders, accordions)
// Make a custom Storyline button keyboard-accessible var btn = document.getElementById("your-button-id"); if (btn) { // Ensure it's focusable btn.tabIndex = 1; btn.setAttribute("role", "button"); btn.setAttribute("aria-label", "Describe what this does"); // Activate on Enter or Space btn.addEventListener("keydown", function(e) { if (e.key === "Enter" || e.key === " ") { e.preventDefault(); btn.click(); // Triggers Storyline's click action } }); // Escape closes any open layer document.addEventListener("keydown", function(e) { if (e.key === "Escape") { var player = GetPlayer(); player.SetVar("CloseLayer", true); } }); }
Drag and Drop — The Hardest One Common Failure
Storyline's built-in drag-and-drop is completely inaccessible to keyboard users. There is no keyboard activation, no focus management, and no screen reader announcements. The only compliant solution is to build a keyboard alternative — either a separate interaction or a custom JS drag-and-drop. See the JS Snippets library for the tab order management code.
05

Screen Reader Text — What Gets Announced

Screen readers announce content in the order it appears in the DOM — which in Storyline is determined by the layer order and the tab order you set. Understanding what gets announced and when is essential for building truly accessible Storyline courses.

What Storyline Announces by Default
Text boxes: Content is read if the object is in the tab order
Buttons: Announced as "button" + the button label
Images: Alt text is read, or "graphic" if no alt text set
Images with "not visible to accessibility tools": Skipped entirely
State changes: NOT automatically announced — must use ARIA live regions
// Announce dynamic changes to screen readers // (e.g. when quiz feedback appears, when score updates) function announceToScreenReader(message) { var region = document.getElementById("sr-announcer"); if (!region) { region = document.createElement("div"); region.id = "sr-announcer"; region.setAttribute("role", "status"); region.setAttribute("aria-live", "polite"); region.setAttribute("aria-atomic", "true"); region.style.cssText = "position:absolute;left:-9999px;width:1px;height:1px"; document.body.appendChild(region); } region.textContent = ""; setTimeout(function() { region.textContent = message; }, 100); } // Call whenever state changes that users need to know about: announceToScreenReader("Correct answer. Your score is now 80 out of 100."); announceToScreenReader("Drag and drop complete. All items placed correctly.");
06

Captions & Transcripts — All Audio and Video

All pre-recorded audio and video content must have captions. Live audio must have real-time captions. Audio-only content (e.g. narration over a static image) must have a transcript. This applies to videos embedded in Storyline, Vyond animations with audio, and recorded narration.

✓ Compliant Approach
  • Add closed captions (SRT or VTT file) to all videos
  • Use Storyline's built-in caption support for narrated slides
  • Provide a transcript download link for audio-only content
  • Caption all Vyond and Camtasia videos before embedding
  • Auto-captions reviewed and corrected before publishing
✗ Non-Compliant
  • Video with no captions at all
  • Auto-captions published without human review
  • Captions as a separate PDF download (not synchronized)
  • "Transcript available on request" — must be immediately available
  • Captions that only show the speaker's name, not the content
